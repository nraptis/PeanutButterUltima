Final Format Example (Current Build)
====================================

Scope
- This is a worked example for the current format constants.
- Checksum words and the archive family id are shown symbolically because they depend on the exact bundled payload and bundle metadata.
- Encryption is still treated as a black box: the plain block shown here is what exists immediately before the block is sealed (encrypted).

Example inputs
- One archive file
- One L3 block per archive
- Archive name: `bundle_demo_0.PBTR`
- Version: `1.7`
- Entries:
  - file `alpha.txt` with `4` bytes of content
  - empty directory `empty_dir`

Example record sizes
- `alpha.txt`
  - path length field: `2`
  - path bytes: `9`
  - content length field: `8`
  - content bytes: `4`
  - total: `23`
- `empty_dir`
  - path length field: `2`
  - path bytes: `9`
  - content length field: `8`
  - total: `19`
- terminator
  - path length field: `2`
  - total: `2`

Logical payload bytes used
- `23 + 19 + 2 = 44`

File size
- archive header: `48`
- one L3 block: `1000704`
- total archive file size: `1000752`

Top-level layout

  bundle_demo_0.PBTR
  +--------------------------------------------------------------+
  | Archive Header (48)                                          |
  +--------------------------------------------------------------+
  | L3 Block 0 (1000704)                                         |
  |  +--------------------------------------------------------+  |
  |  | Recovery Header (48)                                   |  |
  |  +--------------------------------------------------------+  |
  |  | Payload (1000656)                                      |  |
  |  |  alpha.txt record                                      |  |
  |  |  empty_dir record                                      |  |
  |  |  terminator                                            |  |

  +--------------------------------------------------------------+

Notes about the example recovery header
- The skip is `0,0,0` here because the first record starts exactly at this block's logical payload start.
- Recover never uses an all-zero skip as a forward jump target.
- The checksum still validates the block because the checksum covers payload bytes plus the stored skip bytes, even when the skip is all zeros.

Payload byte map

  offset 96
  +--------------------+----------------------------------------------+
  | 96  .. 97          | path_length = 9                              |
  | 98  .. 106         | "alpha.txt"                                  |
  | 107 .. 114         | content_length = 4                           |
  | 115 .. 118         | file bytes                                   |
  | 119 .. 120         | path_length = 9                              |
  | 121 .. 129         | "empty_dir"                                  |
  | 130 .. 137         | content_length = 0xFFFFFFFFFFFFFFFF          |
  | 138 .. 139         | terminator path_length = 0                   |
  | 140 .. 1000751     | zero padding                                 |
  +--------------------+----------------------------------------------+

What happens next
- The writer computes the recovery checksum over the full payload region plus the skip record.
- The writer serializes the recovery header into bytes `48..95`.
- The writer then seals (encrypts) the full `1000704`-byte L3 block before writing it to disk.
