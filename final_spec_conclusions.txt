Final Archive Process Conclusions
=================================

Current conclusions
- The current format is a fixed-size block archive family, not a manifest-plus-data format.
- Each archive file starts with one archive header and then stores a fixed number of L3 blocks.
- Each L3 block carries its own recovery header, checksum, and recovery stride.
- The logical payload stream is simple:
  - path length
  - path
  - content length
  - content bytes
  - terminator

Current operational model
- Archive is deterministic from sorted relative paths.
- Strict unbundle trusts only blocks that:
  - can be read
  - can be unsealed (decrypted)
  - expose a valid recovery header
  - match the stored checksum
  - parse into valid records
- Recover is intentionally less strict:
  - it drops damaged regions
  - it resets parser state after damage
  - it can jump forward using the stored recovery stride after record-parse failures

Directory model
- Only empty directories are stored as standalone records.
- Non-empty directories are recreated implicitly from file paths.

Padding and sizing model
- Unused payload space inside a block is zero-padded before the block is sealed.
- The checksum therefore protects:
  - record bytes
  - terminator bytes
  - any remaining zero padding

Discovery model
- Decode discovery is part of the current format contract.
- The reader depends on:
  - filename parsing for numeric archive indices
  - header family id when readable headers are available
  - dense gap-aware archive-box construction

Practical summary
- The current implementation prioritizes:
  - deterministic write order
  - block integrity checks
  - gap detection in strict mode
  - survivable best-effort recovery in recover mode
