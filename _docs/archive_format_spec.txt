PeanutButter Archive Memory Layout Reverse Specification
Derived from the current codebase on 2026-03-07

Scope
- This document describes the byte layout of the unencrypted archive format.
- This document is intentionally about memory layout, field layout, and block layout.
- This document does not define runtime behavior, validation policy, UI behavior, CLI behavior, or implementation flow.

Important context
- PeanutButterUltima is the standalone unencrypted packer.
- This document is specifically for the unencrypted layout.
- The constants specified in this document, such as "SB_L1_LENGTH" are for demo only, the real constants are different.
- File recovery is made possible by 6-byte recovery-header offsets embedded in the payload stream.
- Archive naming uses `aRecoverySuffix = pRecoveryEnabled ? 'R' : 'X';`
- The `R` suffix indicates an archive marked as recover-able.

Current demo unencrypted-layout constants
- `MAGIC_HEADER_BYTES = 0xF01DAB1E`
- `MAGIC_FOOTER_BYTES = 0xDECAFBAD`
- `MAJOR_VERSION = 1`
- `MINOR_VERSION = 0`
- `SB_PLAIN_TEXT_HEADER_LENGTH = 36`
- `SB_RECOVERY_HEADER_LENGTH = 6`
- `EB_MAX_LENGTH = 16`
- `SB_PAYLOAD_SIZE = 12`
- `SB_L1_LENGTH = 18`
- `SB_L2_LENGTH = 36`
- `SB_L3_LENGTH = 72`
- `MAX_VALID_FILE_PATH_LENGTH = 2048`

Constant relationships
- `SB_L1_LENGTH = SB_PAYLOAD_SIZE + SB_RECOVERY_HEADER_LENGTH`
- `SB_L2_LENGTH = SB_L1_LENGTH + SB_L1_LENGTH`
- `SB_L3_LENGTH = SB_L2_LENGTH + SB_L2_LENGTH`
- `SB_RECOVERY_HEADER_LENGTH = 6`

Archive physical layout
- Each archive file is:
  - one 36-byte archive header
  - followed by zero or more L3 blocks
- Physical file size is therefore:
  - `SB_PLAIN_TEXT_HEADER_LENGTH + (N * SB_L3_LENGTH)`
  - where `N >= 0`

Clear-text header layout
- Header length: 36 bytes
- Byte layout:
  - bytes `0..3`: little-endian 32-bit magic header
  - byte `4`: recovery-enabled flag
  - byte `5`: reserved
  - bytes `6..9`: little-endian 32-bit major version
  - bytes `10..13`: little-endian 32-bit minor version
  - bytes `14..19`: little-endian unsigned 48-bit archive sequence number
  - bytes `20..27`: 8-byte archive identifier
  - bytes `28..31`: 4 reserved bytes
  - bytes `32..35`: little-endian 32-bit magic footer

Clear-text header field values
- Magic header value: `0xF01DAB1E`
- Magic footer value: `0xDECAFBAD`
- Major version value: `1`
- Minor version value: `0`
- Reserved byte 5 is written as `0`
- Archive identifier is stored as 8 raw bytes
- Reserved bytes `28..31` are written as `0`

Payload block hierarchy
- The payload is laid out as repeated L1 blocks.
- Two adjacent L1 blocks form one L2 block.
- Two adjacent L2 blocks form one L3 block.
- Therefore:
  - 4 L1 blocks = 1 L3 block

Current L1 layout
- L1 length: 18 bytes
- L1 byte layout:
  - bytes `0..5`: recovery header block
  - bytes `6..17`: payload bytes

Current L2 layout
- L2 length: 36 bytes
- L2 byte layout:
  - bytes `0..17`: first L1 block
  - bytes `18..35`: second L1 block

Current L3 layout
- L3 length: 72 bytes
- L3 byte layout:
  - bytes `0..17`: first L1 block
  - bytes `18..35`: second L1 block
  - bytes `36..53`: third L1 block
  - bytes `54..71`: fourth L1 block

Recovery header layout
- Recovery header length: 6 bytes
- Stored in bytes `0..5` of every L1 block
- Encoded as little-endian unsigned 48-bit value
- The stored value is an offset quantity used for file recovery

Payload-byte layout inside the L1 block
- Payload-byte length per L1 block: 12 bytes
- Stored in bytes `6..17` of the L1 block
- Logical file-record bytes are packed across these 12-byte payload regions

Logical record layout inside payload bytes
- Each file record is serialized as:
  - `path_length`: 2-byte little-endian unsigned integer
  - `path_bytes`: `path_length` raw bytes
  - `content_length`: 6-byte little-endian unsigned integer
  - `content_bytes`: `content_length` raw bytes

Logical-to-physical mapping
- Logical record bytes are packed only into the payload-byte portions of L1 blocks.
- Recovery-header bytes occupy the first 6 bytes of each L1 block and are not part of file-record payload bytes.
- When a logical record crosses an L1 boundary, its next bytes continue after the next 6-byte recovery header.
- When a logical record crosses an L3 boundary, its next bytes continue in the next sealed L3 block.

Recovery-offset placement
- Every L1 block begins with a 6-byte recovery-header field.
- These fields provide the offset information that makes file recovery possible.
- The offset fields are physically interleaved with file-record payload bytes at fixed 18-byte intervals.

Endian summary
- 32-bit integers are little-endian.
- 48-bit integers are little-endian.
- The 8-byte archive identifier is raw byte storage.
