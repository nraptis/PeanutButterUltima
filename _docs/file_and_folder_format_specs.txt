File and Folder Format Specs

Purpose
- Define the logical record stream used by bundle/unbundle/recover.
- Replace the old manifest-era split format.

Record stream
- The payload is a sequence of logical records:
  - file record
  - folder record (empty directory)
  - end-of-stream marker
- Records are not phase-based. Each record is decoded by its own fields.

File record format
- `[2 bytes] path_length (u16, little-endian)`
- `[path_length bytes] relative_path`
- `[6 bytes] content_length (u48, little-endian)`
- `[content_length bytes] file_contents`

Folder record format
- `[2 bytes] path_length (u16, little-endian)`
- `[path_length bytes] relative_path`
- `[6 bytes] content_length_marker (u48, little-endian)`
- `content_length_marker` must be `0xFFFFFFFFFFFF` (directory marker).
- No content bytes follow for a folder record.

End-of-stream marker
- `[2 bytes] path_length == 0`
- No path bytes and no content-length field follow this marker.
- This marker ends logical decode for unpack.
- Recovery treats this marker as a resync event and continues walking remaining L1 blocks.

Important rule: no file-content padding
- File bytes are exactly `content_length` bytes.
- The implementation must not append `0` or `00` bytes to the end of file contents.
- Any `0/00` values after the last logical record are not file bytes; they are unused payload space.

Recovery header link
- Each L1 block starts with a recovery header:
  - checksum
  - `mDistanceToNextRecord` (distance to next logical record boundary, not file-only).
- Recover walk-ahead uses this distance plus record probing to resynchronize.

Validation notes
- `path_length` must be `1..MAX_VALID_FILE_PATH_LENGTH` for file/folder records.
- `path_length == 0` is reserved for end-of-stream marker.
- `content_length == 0xFFFFFFFFFFFF` means folder record.
- File `content_length` may be `0`.
