Manifest Format Specs

Purpose
- Preserve empty directories during bundle/unbundle without changing regular file records.

Where the manifest lives
- The manifest trailer is serialized after file records in the logical unpack stream.
- Layout:
  - `file records`
  - `0` file-path-length terminator (2-byte little-endian zero)
  - zero or more manifest folder records: `([folder_name_length:u16][folder_name_bytes])*`
  - final `0` manifest-folder-length terminator (2-byte little-endian zero)

Rules
- Only empty leaf folders are stored.
  - Example source:
    - `/apple`
    - `/apple/banana`
  - Stored manifest folder records:
    - `/apple/banana`
- A folder record uses relative path bytes (same path domain as file records).
- Folder paths are materialized with `EnsureDirectory(destination/folder_path)` during unbundle/recover write mode.
- During parse-only validation, folder records are validated but not written.

Manifest integer failures
- `MNF_FNL_001`
  - Trigger: manifest folder path is empty.
- `MNF_FNL_002`
  - Trigger: manifest folder length exceeds remaining bytes in unpack job.
- `MNF_FNL_003`
  - Trigger: manifest folder length exceeds `MAX_VALID_FILE_PATH_LENGTH`.

Compatibility behavior
- Older archives without manifest records remain valid.
- Empty-directory restoration is best-effort for archives that include the manifest trailer.
