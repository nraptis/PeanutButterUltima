Recovery and Unpack Flow (Current Semantics)
============================================

Recovery walk inputs
--------------------
- Recovery builds a discovered archive list from archive headers.
- Archives are normalized and sorted by `archive_index`.
- Only archives with the selected identifier are kept.
- Recovery starts from the selected archive offset in that normalized list.

Recovery walk behavior
----------------------
- Recovery is best-effort and non-fail-fast for data-corruption events.
- The walk continues across every remaining archive from the selected start offset.
- The walk continues across L1 blocks in those archives, even after:
  - bad checksum blocks
  - fence failures (path/content/recovery-distance)
  - zero-length marker / logical end marker
  - end-of-stream while parsing a record
- If logical end is reached before selected payload exhaustion, recovery classifies this as `UNP_EOF_003` (unexpected end of archive) and continues walking.
- Recovery performs page reads/decrypt attempts per L3 page and checksum validation per L1 block while scanning.
- Recovery uses walk-ahead resync:
  - find a valid recovery header + checksum
  - validate recovery distance
  - probe candidate record boundary
  - if valid record boundary is found, resume decode
  - if no resync is found, walk to end-of-set and complete

Recovery start probe
--------------------
- A start probe may locate a candidate record boundary for diagnostics/logging.
- The probe is advisory only; it does not truncate or skip the selected recovery range.
- Recovery decode still starts at the selected archive range start and processes forward.

Unpack behavior
---------------
- Unpack is fail-fast.
- On checksum/fence/read/decrypt issues, unpack stops and returns failure.
- This is intentional to avoid long grind behavior on wrong-password input.
- Premature logical-end classification (`UNP_EOF_003`) is recover-oriented; unpack remains fail-fast.

Notes
-----
- The above "no fatal in recovery walk" semantics apply to archive data integrity/parsing failures.
- Destination filesystem write failures (for example, cannot create or write output paths) are still terminal operation failures.
