Planned Fence Tests

Purpose
- Capture the planned fence-oriented C++ test files.
- Keep the plan aligned with the current record-stream runtime.
- Spell out zero-length corruption behavior for all three length flavors.

Naming rule
- Preferred file naming:
  - `Test_Fences_<CaseName>.cpp`
- Example:
  - `Test_Fences_RecoveryStride_MissingArchive.cpp`

Fence case flavors in code
- Shared test-only definitions live in:
  - `tests/common/Test_Fence_Extras.hpp`
- Flavor structs:
  - `FileNameFenceCase`
  - `FolderNameFenceCase`
  - `FileContentFenceCase`
  - `RecoveryStrideFenceCase`
- Shared probe helper:
  - `TryLengthWithFenceFlag(...)` with `LengthFenceProbeRequest`
- Purpose:
  - Keep generated fence tests in a single shape while varying only case data.

Current format assumptions (must stay true for these tests)
- Logical stream is record-based (no file phase / manifest phase split).
- End-of-stream marker is `path_length == 0` (u16).
- Folder record is identified by content-length marker `0xFFFFFFFFFFFF`.
- Recovery header uses `mDistanceToNextRecord`.

Grouped code mapping
- `FileName_*` => `UNP_FNL_FENCE` unless the mutation creates premature logical end (then `UNP_EOF_003`)
- `FolderName_*` => `UNP_FNL_FENCE` unless the mutation creates premature logical end (then `UNP_EOF_003`)
- `FileContent_*` => `UNP_FDL_FENCE` for fence-class length failures
- `RecoveryStride_*` => `UNP_RHD_FENCE` except `RecoveryStride_Zero` (currently `UNP_EOF_003`)
- `EndOfFile_TrailingGarbage` => `UNP_EOF_002`
- `EndOfFile_TrailingArchives` => `UNP_EOF_001`
- `UnexpectedEnd_*` => `UNP_EOF_003`

Important consistency audit

1. `path_length == 0` is not always a fence failure.
- It is the logical end marker value.
- If it appears before selected payload exhaustion, it is an unexpected-end condition (`UNP_EOF_003`), not `UNP_FNL_FENCE`.

2. `content_length == 0` is not inherently invalid.
- Empty files are valid.
- A zero mutation on content length is only an error when it causes downstream corruption effects, such as:
  - fence failure on subsequent parsed fields
  - premature logical end before selected payload exhaustion (`UNP_EOF_003`)

3. There is no `MAX_VALID_Folder_PATH_LENGTH`.
- Use `MAX_VALID_FILE_PATH_LENGTH` for both file and folder paths.

4. Recovery-distance tests are recover-surface tests.
- `mDistanceToNextRecord` is consumed in recover start probing and recover walk-ahead.
- Unbundle remains fail-fast and is not the primary surface for stride tests.

7. `RecoveryStride_Zero` is a semantic caveat.
- `mDistanceToNextRecord = 0` is legal in the current runtime and can represent "next candidate starts at header end."
- Therefore the zero mutation does not reliably trigger `UNP_RHD_FENCE`.
- Realistic expectation for the current runtime is recover desync/integrity (`UNP_EOF_003`).

5. Folder-name tests are setup-sensitive.
- Seed input must include at least one empty directory.
- Otherwise folder-record mutations are not guaranteed to land on folder records.

6. EOF codes are integrity outcomes, not grouped fence families.
- Keep EOF tests in the plan because they are critical for corrupted archives.

Feasibility legend
- `NOW`: feasible against current runtime behavior
- `RUNTIME_CHANGE`: requires code changes before reliable assertion
- `SETUP_SENSITIVE`: feasible with specific deterministic seed/mutation setup
- `NON_DETERMINISTIC`: mutation may produce one of several valid error outcomes unless constrained

Three zero-length flavors (explicit plan)

Flavor A: file name length mutated to 0
- Canonical test name: `Test_Fences_FileName_Zero_MidPayload.cpp`
- Mutation target: a non-terminal file record `path_length` field.
- Recover expected behavior:
  - detect logical end before selected payload exhaustion
  - classify as `UNP_EOF_003`
  - continue full recover walk through remaining L1 blocks/archives
- Unbundle expected behavior:
  - fail fast at first terminal parse/integrity failure
- Status: `NOW` for plan; may be `SETUP_SENSITIVE` for deterministic offset selection

Flavor B: folder name length mutated to 0
- Canonical test name: `Test_Fences_FolderName_Zero_MidPayload.cpp`
- Mutation target: a non-terminal folder record `path_length` field.
- Recover expected behavior:
  - detect logical end before selected payload exhaustion
  - classify as `UNP_EOF_003`
  - continue full recover walk through remaining L1 blocks/archives
- Unbundle expected behavior:
  - fail fast at first terminal parse/integrity failure
- Status: `SETUP_SENSITIVE` (requires deterministic empty-folder record placement)

Flavor C: file content length mutated to 0
- Canonical test name: `Test_Fences_FileContent_Zero_Desync.cpp`
- Mutation target: a non-terminal file record `content_length` field.
- Logical possibilities:
  - valid empty-file reinterpretation followed by continued parse
  - downstream parse fence (`UNP_FNL_FENCE` or `UNP_FDL_FENCE`)
  - premature logical end before payload exhaustion (`UNP_EOF_003`)
- Recover expected behavior:
  - never stop scanning at first corruption effect
  - if early logical stop is observed before payload exhaustion, include `UNP_EOF_003`
- Unbundle expected behavior:
  - fail fast on first surfaced failure
- Status: `NON_DETERMINISTIC` unless mutation location is tightly constrained
