PeanutButterUltima Testing Requirements
=======================================

Purpose
-------

This file captures how the new testing system should work.

It is intentionally higher level than implementation detail.
It is meant to constrain the architecture of the test harness before the
new runtime is built out.


Core testing philosophy
-----------------------

The tests are not a second implementation of the product.

The tests should:

- drive the real app entry points
- observe outputs
- compare outputs against semantic expectations

The tests should not:

- rebuild the storage format logic in parallel
- invent a second archive parser as a source of truth
- force production code into odd dependency patterns just to make tests pass


Required test architecture
==========================

1. Hard-drive abstraction
-------------------------

All hard-drive writes must route through one mirror-able entry point.

Meaning:

- production/runtime code should not write directly to the real filesystem
  in arbitrary places
- there should be one coherent filesystem service boundary
- all file creation, overwrite, directory creation, enumeration, rename,
  delete, and byte reads/writes should route through that boundary

Production mode:

- the boundary talks to the real hard drive

Test mode:

- the boundary talks to a mock hard drive
- the mock hard drive stores files and directories in memory

Design consequence:

- a "pack job" or "unpack job" should be runnable against either:
  - real filesystem service
  - in-memory fake filesystem service

This is the main injection point for tests.


2. Logger abstraction
---------------------

The tests export logs to a dummy logger.

Meaning:

- runtime code should log through one logger interface
- tests should be able to inject a no-op logger or capturing logger
- tests may assert on log messages when useful
- tests should not depend on stdout/stderr scraping

Preferred logger modes:

- `DummyLogger`
- `CapturingLogger`


3. Test-owned constants
-----------------------

Tests get their own version of important constants.

Meaning:

- tests must be able to specify archive size and other important runtime
  knobs without mutating global runtime state
- test constants should live in a dedicated test config object or test
  runtime settings object
- production constants and test constants must be clearly separated

Important:

- tests should configure runtime behavior through explicit settings
- tests should not reach into production globals and patch them in place

Examples of settings that tests may need to control:

- archive file length
- archive block count
- path/naming defaults when relevant


4. No second source of truth
----------------------------

The new tests must not maintain a second implementation of product truth.

This is a hard rule.

Specifically:

- no blind archive encoder used as oracle
- no blind archive decoder used as oracle
- no second implementation of recovery-header interpretation used as oracle
- no circular expectation generation from product output bytes

Allowed:

- direct semantic assertions
- small byte-level fixtures
- invariant checks on specific known fields
- expected file trees
- expected archive names
- expected sequence-number behavior

Not allowed:

- "implementation A matches implementation B" as the primary testing style


5. Common test utilities
------------------------

Tests should route through shared test utilities.

Meaning:

- repeated testing operations should live in a common test support layer
- the support layer should help with setup and assertion
- the support layer should not become a second implementation of the app

Examples of acceptable common utilities:

- create fake filesystem with input tree
- run pack job through app entry point
- run unpack job through app entry point
- run recover job through app entry point
- run test execution through app entry point
- compare tree `input` vs tree `unarchived`
- compare tree `input` vs tree `recovered`
- collect archive files from fake filesystem
- assert archive names
- assert archive header fields

Example requirement:

- there should be one common utility for:
  - "are files in `input` on the fake hard drive equal to files in
    `unarchived` on the fake hard drive?"


App entry point requirement
===========================

To run a pack job, the tests should hit the app entry point.

Meaning:

- tests do not bypass the main runtime flow
- tests do not instantiate deep internal helpers as the primary interface
- tests should be able to call something equivalent to:
  - `RunPackJob(...)`
  - `RunUnpackJob(...)`
  - `RunRecoverJob(...)`

Each such app entry point must accept:

- filesystem service
- logger
- runtime settings

So the test harness can inject:

- fake hard drive
- dummy or capturing logger
- test-specific settings


Recommended runtime shape for testing
=====================================

The exact names can change, but the shape should be roughly:

- `AppContext`
  - filesystem
  - logger
  - settings

- `RunBundle(AppContext&, BundleRequest)`
- `RunUnbundle(AppContext&, UnbundleRequest)`
- `RunRecover(AppContext&, RecoverRequest)`
- `RunValidate(AppContext&, ValidateRequest)`

The key point:

- tests should configure the world once through context
- then call the same top-level flows as the UI


What the tests should assert
============================

Unencrypted-only rule
---------------------

PeanutButterUltima is the standalone unencrypted packer.

Meaning:

- this sub-project does not test encryption
- test naming should not carry `NoEncryption` as a category label
- the semantic contract is the unencrypted archive format only
- the injected `Crypt` seam still exists as an architectural boundary, but
  the test suite does not use it to add a second mode matrix

Pack / unpack
-------------

Primary semantic assertions:

- input file tree equals unpacked file tree
- archive names are correct
- clear-text header recovery flag matches `R` / `X`
- archive sequence numbers are correct
- archive UUID grouping is correct where relevant
- invalid/corrupt archive scenarios fail the correct way

Recovery
--------

Primary semantic assertions:

- recovery output contains the files it should contain
- invalid recovery starts are rejected
- recoverable starts produce the expected resumed file stream

Validation
----------

Primary semantic assertions:

- equal trees pass
- unequal trees fail
- mismatches are reported meaningfully


What the tests should stop doing
================================

The old-style tests should not be carried forward as-is if they rely on:

- memory archive writer/provider stacks that bypass the app entry point
- runtime global mutation for archive sizing
- duplicate archive parsers/encoders as truth engines
- circular recovery expectation generation from produced archives

Non-goals
=========

The new testing system does not need:

- dependency injection everywhere
- mocks for every internal helper
- a parallel in-memory archive format implementation

It does need:

- one filesystem boundary
- one logger boundary
- one app-entry-point boundary
- shared test utilities
- semantic assertions


Final rule
==========

If a proposed test helper starts to feel like a reimplementation of the
product, it is probably the wrong shape.

The tests should bend around the mainline app entry points and the
filesystem/logger seams, not force the runtime to bend around test-only
inventions.
