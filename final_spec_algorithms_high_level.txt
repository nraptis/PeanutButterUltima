Final Archive / Unarchive / Recover Specification (High Level)
==============================================================

Scope
- This file summarizes the current archive, unarchive, and recover flow.
- Encryption is treated as a black box. The archive path seals (encrypts) L3 blocks, and the decode path unseals (decrypts) L3 blocks.

Global rules
- Source entries are processed in sorted relative-path order.
- The logical record stream is written across fixed-size L3 blocks.
- Every L3 block has its own recovery header.
- Strict unbundle is fail-fast.
- Recover is best-effort and block-oriented.

Archive
1. Collect source entries.
  - Single file input becomes one file record under its basename.
  - Directory input contributes all files recursively.
  - Empty directories become explicit directory records.
2. Sort entries by relative path.
3. Build the logical record stream:
  - `u16 path_length`
  - path bytes
  - `u64 content_length`
  - file bytes for file records
  - empty-directory marker for empty folders
  - final terminator record with `path_length = 0`
4. Compute archive count from total logical bytes and configured blocks per archive.
5. For each archive file:
  - write the archive header
  - for each L3 block:
    - zero the block
    - fill payload bytes from the logical stream
    - compute recovery stride to the next record start
    - compute checksum
    - write the recovery header
    - seal (encrypt) the full L3 block
    - append the block to the archive file

Unarchive
1. Collect candidate files from the chosen file or folder.
2. Select one archive family by filename template and, when possible, header family id.
3. Build a dense archive index window, including empty boxes for missing indices.
4. Decode block-by-block in strict mode:
  - read block
  - unseal (decrypt) block
  - parse recovery header
  - verify checksum
  - parse records
  - create folders and write files
5. Succeed only after the terminator record is reached.

Recover
1. Use the same candidate-discovery pipeline as unarchive.
2. Decode block-by-block in recover mode.
3. On missing or damaged blocks:
  - count the block as failed
  - abort any partial output file from the damaged region
  - reset the parser
  - continue
4. On record-parse failure inside an otherwise healthy block:
  - abort any partial output file
  - reset the parser
  - try the recovery stride stored in that block
  - if the stride is invalid, continue with the next block
5. Succeed when:
  - the terminator record is reached, or
  - archive space is exhausted after at least one file was recovered

Important current behaviors
- Only empty directories are stored as directory records.
- Non-empty directories are recreated from file paths during decode.
- Strict unbundle stops on the first gap, short block, unseal failure, checksum failure, or record-parse failure.
- Recover can skip past damage, but it does not guarantee a complete reconstruction.
