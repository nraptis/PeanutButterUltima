PeanutButterUltima File Format
==============================

Scope
- This file explains the current archive file format in plain language.
- It matches the checked-in bundle, unbundle, and recover code.
- Encryption is a black box here:
  - archive: seal (encrypt) each L3 block
  - decode: unseal (decrypt) each L3 block

Current constants
- Magic: `0xDECAFBAD`
- Version: `1.7`
- Archive header: `48` bytes
- Recovery header: `48` bytes
- L3 block: `1000704` bytes
- Payload bytes per L3 block: `1000656` bytes
- Max path length: `2048`
- Max archive count: `65535`
- Max blocks per archive: `2048`

Byte order
- All integers are little-endian.

Big picture

  one archive file
  +--------------------------------------------------------------+
  | Archive Header (48)                                          |
  +--------------------------------------------------------------+
  | L3 Block 0 (1000704)                                         |
  |  +--------------------------------------------------------+  |
  |  | Recovery Header (48)                                   |  |
  |  +--------------------------------------------------------+  |
  |  | Payload (1000656)                                      |  |
  |  +--------------------------------------------------------+  |
  +--------------------------------------------------------------+
  | L3 Block 1 (1000704)                                         |
  |  +--------------------------------------------------------+  |
  |  | Recovery Header (48)                                   |  |
  |  +--------------------------------------------------------+  |
  |  | Payload (1000656)                                      |  |
  |  +--------------------------------------------------------+  |
  +--------------------------------------------------------------+
  | ...                                                          |
  +--------------------------------------------------------------+

The archive header

  bytes 0..47

  +--------+----------------------+----------------------------------+
  | Offset | Field                | Meaning                          |
  +--------+----------------------+----------------------------------+
  | 0      | magic                | must be 0xDECAFBAD              |
  | 4      | version_major        | must be 1                       |
  | 6      | version_minor        | must be 7                       |
  | 8      | archive_index        | zero-based index of this file   |
  | 12     | archive_count        | total files in the archive set  |
  | 16     | payload_length       | block region bytes after header |
  | 20     | record_count_mod256  | record starts in this archive   |
  | 21     | folder_count_mod256  | empty-dir starts in this archive|
  | 22     | encryption_strength  | 1=high, 2=medium, 3=low         |
  | 23     | reserved8            | currently 0                     |
  | 24     | reservedA            | currently 0                     |
  | 32     | reservedB            | currently 0                     |
  | 40     | archive_family_id    | family identifier               |
  +--------+----------------------+----------------------------------+

Important note about `payload_length`
- The name is misleading.
- Current writer value:
  - `payload_length = block_count * 1000704`
- So it counts the full serialized L3 block region after the archive header.
- It does not count only the logical record payload bytes.

One L3 block

  +--------------------------------------------------------------+
  | Recovery Header (48)                                         |
  +--------------------------------------------------------------+
  | Payload (1000656)                                            |
  |  logical records                                              |
  |  terminator                                                   |
  |  zero padding                                                 |
  +--------------------------------------------------------------+

The recovery header

  bytes 0..47 within each L3 block

  +--------+-------------------------+-----------------------------+
  | Offset | Field                   | Meaning                     |
  +--------+-------------------------+-----------------------------+
  | 0      | checksum.word1          | block integrity word 1      |
  | 8      | checksum.word2          | block integrity word 2      |
  | 16     | checksum.word3          | block integrity word 3      |
  | 24     | checksum.word4          | block integrity word 4      |
  | 32     | checksum.word5          | block integrity word 5      |
  | 40     | skip.archive_distance   | forward archive delta       |
  | 42     | skip.block_distance     | forward block delta         |
  | 44     | skip.byte_distance      | forward payload-byte delta  |
  +--------+-------------------------+-----------------------------+

What the checksum protects
- The checksum is calculated from:
  - the whole plain payload region of the block
  - the full 8-byte skip record
- The checksum does not include the recovery header itself.
- Because the block is zeroed before payload bytes are filled, the checksum also protects the trailing zero padding in the payload region.

What the skip record means
- The writer looks at the current block's logical payload start.
- It finds the next record start at or after that point.
- It encodes the forward distance as:
  - archive distance
  - block distance
  - byte distance

Conceptually:

  current block payload start
  v
  [........current block payload........][next record begins here]
   \__________________ skip distance _____________________________/

Recover only uses the skip record after a record-parse failure in a block that already:
- could be read
- could be unsealed (decrypted)
- passed checksum

The logical record stream

  each record
  +----------------------+
  | u16 path_length      |
  +----------------------+
  | path bytes           |
  +----------------------+
  | u64 content_length   |
  +----------------------+
  | file content bytes   |  file record only
  +----------------------+

Special cases
- `path_length == 0`
  - terminator record
  - ends the logical stream
- `content_length == 0xFFFFFFFFFFFFFFFF`
  - empty directory record
  - no file content bytes follow

Directory behavior
- Only empty directories get their own records.
- Non-empty directories are rebuilt implicitly from file paths when files are written back out.

Path rules at decode time
- Paths must be safe relative paths.
- The decoder rejects:
  - absolute paths
  - `.` path segments
  - `..` path segments
  - control characters
  - paths longer than `2048` bytes

Worked layout example

Assume one archive file with one L3 block and these records:
- file `alpha.txt` with 4 bytes
- empty directory `empty_dir`
- terminator

The file looks like this:

  absolute file offsets

  0       47       Archive Header
  48      95       Recovery Header
  96      118      Record: alpha.txt
  119     137      Record: empty_dir
  138     139      Terminator
  140     1000751  Zero padding

Zoomed in:

  +--------------------+----------------------------------------------+
  | 96  .. 97          | path_length = 9                              |
  | 98  .. 106         | "alpha.txt"                                  |
  | 107 .. 114         | content_length = 4                           |
  | 115 .. 118         | 4 file bytes                                 |
  | 119 .. 120         | path_length = 9                              |
  | 121 .. 129         | "empty_dir"                                  |
  | 130 .. 137         | content_length = 0xFFFFFFFFFFFFFFFF          |
  | 138 .. 139         | terminator path_length = 0                   |
  | 140 .. 1000751     | zero padding                                 |
  +--------------------+----------------------------------------------+

How archive names are written
- Writer format:
  - `prefix + source_stem + "_" + zero_padded_index + suffix`
- Example:
  - `bundle_demo_0.PBTR`
  - `bundle_demo_01.PBTR`
  - `bundle_demo_002.PBTR`
- Zero-padding width comes from the number of digits in `archive_count - 1`.

How archive names are parsed
- The decoder reads the trailing digit run before the extension as the numeric archive index.
- It does not require an underscore before the digits.

Strict unarchive vs recover
- Strict unarchive stops on the first missing archive slot, short block, unseal failure, checksum failure, or record-parse failure.
- Recover skips past damaged regions when it can and may succeed without ever reaching the terminator, as long as it managed to recover at least one file before archive space ran out.
