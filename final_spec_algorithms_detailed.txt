Final Archive / Unarchive / Recover Specification (Detailed)
============================================================

Notation
- `ARCHIVE_HEADER_LENGTH = 48`
- `RECOVERY_HEADER_LENGTH = 48`
- `BLOCK_SIZE_L3 = 1000704`
- `PAYLOAD_BYTES_PER_L3 = 1000656`
- `payload_bytes_per_archive = archive_block_count * 1000656`

A) Archive

A1. Discovery and planning
1. Validate request:
  - destination directory must be non-empty
  - archive block count must be in `[1, 2048]`
2. Receive source entries from the caller.
3. Sort source entries by `mRelativePath`.
4. Validate each entry:
  - relative path must be non-empty
  - relative path length must be `<= 2048`
  - file entries must expose a readable file length
5. Build the logical record stream size:
  - every record contributes `2 + path_length + 8 + content_length`
  - empty-directory records use `content_length = 0xFFFFFFFFFFFFFFFF` and no content bytes
6. Record every record start offset.
7. Append one extra record-start offset for the terminator.
8. Add the terminator size (`2` bytes) to total logical bytes.
9. Compute:
  - `archive_count = ceil(total_logical_bytes / payload_bytes_per_archive)`
  - minimum archive count is `1`
10. Reject if `archive_count > 65535`.
11. For each archive ordinal:
  - compute the archive filename
  - count how many record starts fall inside that archive's logical payload window
  - count how many empty-directory record starts fall inside that window
  - store `record_count % 256` and `folder_count % 256`
12. Compute the archive family id from bundle metadata.

A2. Archive writing
1. Ensure the destination directory exists.
2. Build a streaming logical-record reader.
3. For each planned archive:
  - open the output file
  - write the `48`-byte archive header
4. For each L3 block in that archive:
  - zero the full `1000704`-byte plain block
  - stream logical bytes into the payload region `[48 .. 1000703]`
  - compute the current block's logical payload start
  - find the next record start at or after that logical offset
  - convert that distance into the fixed-size skip record
  - compute the checksum over:
    - the full payload region
    - the skip record
  - serialize the `48`-byte recovery header into the block prefix
  - seal (encrypt) the full L3 block
  - write the full L3 block to the archive file
5. After all planned archives are written, require that the logical stream has terminated.

A3. Directory handling
- Single-file bundle input stores one file record using the source file's basename.
- Directory bundle input stores:
  - all files recursively
  - only empty directories as explicit directory records
- Non-empty directories are not stored as standalone records.

B) Unarchive and Recover shared discovery

B1. Candidate collection
1. If the chosen source path is a file, use that one file.
2. If the chosen source path is a directory:
  - collect top-level files
  - if no top-level files exist, fall back to a recursive scan
3. Sort candidate paths.

B2. Candidate parsing
1. For each candidate:
  - open it
  - record file length
  - parse `(prefix, numeric_index, suffix, digit_width)` from the filename
  - if enough bytes exist, try to read and validate the archive header
  - infer readable block count from file length and, if possible, header `mPayloadLength`
2. Ignore candidates whose filenames do not parse.

B3. Family selection
1. Sort parsed candidates by path.
2. Choose the first parsed filename as the template anchor.
3. Keep only candidates whose parsed prefix and suffix match that anchor.
4. If readable headers exist inside that anchor family:
  - vote by `mArchiveFamilyId`
  - keep only the dominant family id
5. De-duplicate by numeric index:
  - prefer a candidate with a readable header
  - otherwise prefer the longer file
6. Sort surviving indices.
7. Clip the selection window to:
  - `[min_index, min_index + MAX_ARCHIVE_COUNT - 1]`
8. Count missing indices as gaps.
9. If readable headers vote for a longer tail via `mArchiveCount`, extend the index window and mark those missing tail indices as gaps too.
10. Build a dense `ArchiveFileBox` list from `min_index` through `max_index`.

C) Unarchive and Recover shared block loop

C1. Decode setup
1. Ensure destination directory exists.
2. If encryption is enabled:
  - resolve a single encryption strength from readable headers
  - build the unseal (decrypt) engine
3. Determine `blocks_per_archive`:
  - first choice: first readable header whose `mPayloadLength` is a positive multiple of `BLOCK_SIZE_L3`
  - fallback: discovered block count from the first archive
  - final fallback: `1`
4. Recompute archive-box payload spans using that `blocks_per_archive`.
5. Initialize the parser at record boundary.

C2. Per-block strict flow
1. For the current `(archive_index, block_index, payload_offset)`:
  - fail if the archive box is empty
  - fail if the archive file cannot be opened
  - fail if the archive is too short for the requested block
  - fail if the block cannot be read
  - unseal (decrypt) the block
  - fail if unseal fails
  - parse the recovery header
  - fail if recovery-header parse fails
  - recompute checksum
  - fail if checksum mismatches
  - feed payload bytes into the record parser
  - fail if record parsing fails
2. Succeed only when the parser reaches the terminator record.
3. Fail if archive space is exhausted before the terminator.

C3. Record parser
1. Read `u16 path_length`.
2. If `path_length == 0`, terminate successfully.
3. Read `path_length` bytes and validate the path:
  - relative only
  - no `.` or `..`
  - no control bytes
  - length `<= 2048`
4. Read `u64 content_length`.
5. If `content_length == 0xFFFFFFFFFFFFFFFF`:
  - ensure the output directory exists
  - count one created folder
  - reset parser state
6. Otherwise:
  - reject implausible lengths above `1 << 40`
  - ensure the output file's parent directory exists
  - open the output file
  - stream exactly `content_length` bytes into it
  - close the file
  - count one written file
  - reset parser state

D) Recover-only differences

D1. Recover handling of damaged blocks
- If the current archive box is empty:
  - count a failed region
  - abort any partial output file
  - reset parser state
  - advance to the next archive
- If the archive file cannot be opened, is too short, cannot be read, fails unseal, fails recovery-header parse, or fails checksum:
  - count one failed block
  - abort any partial output file
  - reset parser state
  - advance to the next block

D2. Recover handling of record-parse failures
1. Count one failed block.
2. Abort any partial output file.
3. Reset parser state.
4. Try to resolve the stored recovery stride:
  - reject all-zero stride
  - reject `block_distance >= blocks_per_archive`
  - reject `byte_distance >= 1000656`
  - reject missing target archive
  - reject target block outside the target archive's readable block count
5. If stride resolution succeeds:
  - jump directly to the target archive, block, and payload byte offset
6. Otherwise:
  - advance to the next block

D3. Recover success and failure
- Recover returns success if:
  - it reaches the terminator, or
  - it exhausts archive space after recovering at least one file
- Recover returns `RECOVER_EXHAUSTED` if archive space is exhausted before the terminator and no files were recovered.

E) Archive naming and index parsing
- Writer format:
  - `prefix + source_stem + "_" + zero_padded_index + suffix`
- Reader format:
  - parse any trailing digit run before the file extension as the archive index
- Decode family selection depends on filename parsing as well as, when available, header family id.
