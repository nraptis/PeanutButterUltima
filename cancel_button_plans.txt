Cancel Button Plans
===================

Goal
----
Design a cancellation policy that feels predictable to users:
1) User clicks cancel.
2) System tries up to 10 seconds to finish the current file.
3) If not finished, log the file that was interrupted.
4) Delete that partial file.
5) Exit cleanly.
6) If no file stream is open, cancel at the next check immediately.


Current Behavior (What the code does now)
-----------------------------------------
- UI cancel sets an atomic flag (`mCancelRequested`) in `main_qt.cpp`.
- Bundle/Decode check cancel only at top-level entry points before/after some large phases.
- Core loops in Bundle, Decode, Recover, and Validate do not currently receive a cancel token.
- Validate currently has no active cancel path in core execution.
- Partial output handling:
  - Decode/Recover uses `PartialFileWriter`, which can `AbortFile()` (close stream).
  - `AbortFile()` does not delete the partial file.
- FileSystem interface currently has no single-file delete operation (only directory clear).

Bottom line: current cancel is "best effort before starting" rather than true in-flight cancellation.


Can the requested policy be supported?
--------------------------------------
Yes, but not as a tiny patch. It needs deliberate plumbing and a few API additions.

What already fits:
- Cooperative cancel checks in loops are a good fit for this codebase.
- Bundle and Decode already have clean loop boundaries (archive/block/file) where checks can be inserted.
- Recover already has structured retry loops that can host checks.

What is missing:
- A cancel token/context passed into core functions.
- A way to delete a single partial file.
- Tracking/logging of "current input file" and "current output file" at interruption time.
- A real "10-second grace window" state machine.


Biggest Risks
-------------
1) 10-second guarantee is not hard-real-time.
- If the thread is blocked in filesystem I/O, the check cannot run until the call returns.
- So policy should be "target 10 seconds when cooperative points are reached," not a strict deadline.

2) Partial file cleanup cannot be correct without explicit path tracking.
- Need to know exactly which file is active when cancel escalates.
- Wrong-path deletion is the highest-severity risk.

3) Large discovery calls are currently coarse.
- `ListFilesRecursive` can take a long time before control returns.
- Cancel responsiveness during huge directory scans may still be delayed unless listing is made incremental.

4) Recover complexity.
- Recover alternates decode/retry/skip-walk states.
- Cancel must not leave misleading error classification vs user-canceled state.

5) Validate mode mismatch.
- User expects all active modes to cancel.
- Validate currently has no in-core cancel handling and can appear "uncancelable."


Cancel Policy That Fits Cleanly
-------------------------------
Use a two-stage cooperative policy:

Stage A: Soft cancel (grace period)
- Triggered immediately when user clicks cancel.
- Record deadline `now + 10s`.
- Continue until a "safe boundary" while trying to finish current file.

Stage B: Hard cancel (after deadline)
- If current file still active when deadline passes:
  - log interrupted file path(s),
  - close stream,
  - delete partial output file,
  - return canceled result.

Safe boundaries by mode:
- Bundle:
  - Prefer boundary: end of current archive file.
  - If deadline expires mid-archive, close and delete current archive path.
- Unbundle:
  - Prefer boundary: end of current decoded output file.
  - If deadline expires, abort partial writer and delete active output file.
- Recover:
  - Same as Unbundle for active output file.
  - If in skip/discovery phases with no open output file, cancel immediately at next check.
- Validate:
  - No output streams, so cancel on next check (file-comparison loop boundary).


Implementation Plan
-------------------
1) Add cancellation context for core operations.
- Pass a `CancelContext` (or equivalent) from UI engine into:
  - `PerformBundleFlight`
  - `BuildDecodePlan` and `DecodeCore` call chain
  - `RunSanity`

2) Add FileSystem single-file delete API.
- Add `RemoveFile(path)` to FileSystem + LocalFileSystem + mocks.
- Optional: add `RemoveEmptyParents` helper later (not required initially).

3) Track active file paths.
- Bundle:
  - track current source entry and current archive output path.
- Decode/Recover:
  - store active output path in `PartialFileWriter` (getter for active path).

4) Insert cooperative checks at natural boundaries.
- Bundle:
  - discovery loop, archive loop, block loop.
- Decode:
  - candidate-scan loop, archive loop, block loop, parser file transitions.
- Recover:
  - per-attempt loop and invalid-walk transitions.
- Validate:
  - per-file compare loop and periodic byte loop checks.

5) Implement grace + hard-cancel behavior.
- On first observed cancel: set deadline.
- Until deadline: keep trying to reach file boundary.
- After deadline:
  - if active file exists, log + delete partial file, then cancel.
  - if no active stream, cancel immediately.

6) Logging contract.
- Add explicit cancellation logs:
  - `[Mode] Cancellation requested.`
  - `[Mode] Finishing current file '{path}' before cancellation (up to 10 seconds).`
  - `[Mode] Cancellation deadline reached; deleting partial file '{path}'.`
  - `[Mode] Cancellation completed.`

7) Tests.
- Unit tests for:
  - cancel before start
  - cancel during active file write (finish within grace)
  - cancel during active file write (deadline exceeded, partial deleted)
  - cancel during non-stream phases (immediate)
  - validate cancel responsiveness
  - recover cancel in skip-walk and in file-write phase


Practical Recommendation
------------------------
Implement this in two passes:

Pass 1 (lower risk, quick value):
- Cooperative checks in all major loops.
- Basic canceled result propagation in all modes.
- Logging for requested/completed cancellation.

Pass 2 (full UX policy):
- 10-second grace window.
- Active file path tracking.
- Partial file deletion with new FileSystem API.

This sequencing reduces regression risk while moving steadily toward the exact user-facing behavior.
